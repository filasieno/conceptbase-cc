Anleitung zur Benutzung der ConceptBase-SourceCode-Verwaltung

1. Überblick über die Befehle

Der ConceptBase-Pool basiert auf CVS, das cvs-Kommando kann also mit seinen üblichen Optionen auf den Pool angewendet werden. Hier kurz eine Aufstellung der wichtigsten Befehle mit Angabe der alten Befehle, die $auml;hnlich dazu sind.
  • cvs commit Einchecken einer Datei (CB_CheckIn geht auch noch)
  • cvs checkout Auschecken einer Datei bzw. eines kompletten Directories (CB_CheckOut geht auch noch). Im Gegensatz zum alten CB_CheckOut werden aber keine Sperren für andere Benutzer gesetzt (evtl. gibt es also einen Konflikt beim Einchecken). Zu beachten ist, dass man bei checkout den kompletten Pfad zur Datei angeben muss (inkl. ProductPOOL, z.B. cvs checkout ProductPOOL/serverSources/Prolog_Files/Literals.pro ) und dass man sich im Wurzelverzeichnis seines lokalen Pools befinden muss (d.h. dort wo das ProductPOOL-Verzeichnis ist). Fuer einzelne Dateien duerfte daher das alte CB_CheckOut Kommando einfacher sein.
  • cvs updateUpdate des lokalen Pools durch neuere Dateien im CVS-Repository. D.h. wenn jemand eine neue Version einer Datei in das CVS-Repository eingespielt hat, die man vorher mit checkout geholt hatte, wird sie durch update aktualisiert. Wenn man lokal Aenderungen an der Datei gemacht hat, werden die beiden Versionen der Dateien zusammengefuehrt (Merging). Die Unterschiede werden dann in der Datei hervorgehoben (vergleichbar mit diff).
  • cvs add Anlegen einer Datei (ehemals CB_CreateFile)
  • CB_CreateCVSPool Anlegen eines Pools (ehemals CB_CreatePool)
  • cvs log Anzeigen der Versionsgeschichte einer Datei (ehemals CB_FileInfo)
  • CB_Make Make-Kommando für CB-Pool
Um den ConceptBase-CVS-Pool nutzen zu können, muss man die Umgebungsvariable CVSROOT auf einem der folgenden Werte setzen:
  • :local:/home/cbase/CVS bzw. /home/cbase/CVS
    wenn man CVS unter Solaris benutzt und das cbase-Homeverzeichnis unter /home/cbase gemountet ist (Standard für i5-Solaris-Rechner)
  • :ext:user@host:/home/cbase/CVS
    wenn man CVS remote über SSH benutzen will. Zusätzlich sollte man die Umgebungsvariable CVS_RSH auf die zu benutzende SSH setzen, z.B. /usr/bin/ssh . host sollte hier ein Solaris-Rechner des I5-Netzes sein, der /home/cbase gemountet hat (z.B. manet, warhol oder dix) und user sollte ein gültiger Benutzername auf diesem Rechner sein. Ausserdem sollte man SSH mit den Public-Keys so einstellen, dass man sich ohne Passwort einloggen kann. Evtl. muss man noch die Umgebungsvariable CVS_SERVER setzen. Diese Variable gibt an, welche Programm auf Server ausgeführt werden soll (normalerweise cvs). Das ist vor allem dann notwendig, wenn cvs nicht im PATH enthalten ist.

Im CVS-Repository stehen zwei Pools zur Verfuegung, einer fuer ConceptBase (ProductPOOL) und einer fuer die Programme der SourceCode-Verwaltung (AdminPOOL), wobei im AdminPOOL auch noch die alten Programme zum Checkin/Checkout vorhanden sind.

Wenn man mit der Kommandozeilenversion von cvs Probleme hat, kann man auch ein GUI benutzen. Fuer Windows sollte man WinCVS benutzen, dort gibt es auch Links zu anderen GUI-Clients fuer CVS.

CVS ist aber relativ einfach zu benutzen. Die wichtigsten Kommandos sind oben erwähnt (checkout, commit, update) und auch von der Kommandozeile aus leicht zu bedienen. Eine gute Uebersicht ueber alle CVS-Befehle gibt die CVS Quick Reference.

Es gibt auch einen Read-Only Webzugriff: http://www-i5.informatik.rwth-aachen.de/cgi-bin/cvsweb/cvsweb.cgi/. Benutzername ist "cbase" und Passwort ist "TheBest".

Ausserdem werden folgende Environment-Variablen für die Benutzung eines interaktiven Servers benoetigt:

setenv CB_WORK <lokaler Pool>

Z.B. setenv CB_WORK /home/quix/CB/ProductPOOL/

Beim Start eines interaktiven Servers wird zunaechst in dem Pool nach lokalen Dateien gesucht, der in dieser Variable angegeben ist.

Um Dateien in der SourcePool Verwaltung bearbeiten zu koennen sind im groben folgende Schritte erforderlich:

  1. Anlegen eines neuen lokalen Pools unter dem privaten Benutzer mit cvs co "directory" (z.B. cvs co ProductPOOL/serverSources/Prolog_Files)
  2. Anlegen der entsprechenden Komponenete mit CB_Make (evtl. mehrfach)
  3. CB_CheckOut , CB_CheckIn und CB_CreateFile zum Ändern und Anlegen von Dateien

Beispiel: GlobalParameters.pro soll geaendert werden

CB_CreateCVSPool

setenv CB_WORK /home/username/CB/ProductPOOL

cd $CB_WORK

CB_Make info

#Jetzt wird eine Menge Information uber das aktuelle Verzeichnis ausgegeben.

# Unter anderem, welche Komponenten man anlegen kann: Sub-Module : bin doc examples lib man serverSources i86pc sun4

# man moechte einen Server-Quelltext bearbeiten (GlobalParameters.pro):

CB_Make serverSources

# jetzt wird das Verzeichnis serverSources in allen Architekturvarianten angelegt.

# GlobalParameters ist architekturunabhaengig, also wechseln nach serverSources (und nicht z.B. sun4/serverSources)

cd serverSources

CB_Make info

#Ausgabe u.A. Sub-Module : Prolog_Files C_Files

# Prolog-Sourcen :

# hier liegt also noch nicht die gesuchte Datei (keine Sourcen vorhanden), aber im Modul Prolog_Files

CB_Make Prolog_Files

cd Prolog_Files

CB_CheckOut GlobalParameters.pro

#aendern der Datei und Testen:

CBserver -i test -d ~/test3.99

# scheint zu funktionieren: .... >>> : RESPONSE IS CBserver#17555078.0 {ok}

# einchecken

CB_CheckIn GlobalParameters.pro

#fertig

Zu Beachten

  • Generell ist wohl sinnvoll sich an die CVS Kommandos zu gewöhnen und nicht die alten Befehle CB_CheckIn/CB_CheckOut zu benutzen, da das Arbeiten mit CVS anders ist als mit SCCS (z.B. keine expliziten Sperren).
  • nie in das Verzeichnis src wechseln, dann folgt man naemlich einem link und steht ploetzlich unter der ~CBase-Hierarchie und nicht mehr zuhause im lokalen Pool
  • Bug reports
  • don't panic

CB_Make

CB_Make ist ein kurzes Shell-Skript, das einige Environment-Variablen setzt und dann das GNU-Make aufruft. Man kann im Source-Pool nicht make oder gmake benutzen.

CB_Make liest immer das Makefile des aktuellen Verzeichnisses und config.mk aus der Wurzel des ProductPOOLs. Meistens wird auch rules.mk eingelesen, aber das ist nicht immer noetig. rules.mk und config.mk enthalten Default-Werte und Regeln, die fuer den ganzen Pool gelten. Ein Fehler beim Make-Aufruf sollte also immer in diesen drei Dateien zu finden sein, die Default-Makefiles von GNU-Make werden nicht benutzt.

Falls die generierten Dateien variantenabhaengig sind (z.B. *.o, *.wic ), so findet man diese wieder im entsprechenden Variantenverzeichnis (z.B. Wic-Files in sun4/serverSources/Prolog_Files/objects )

Zur Zeit haben wir noch keinen variantenabhaengigen Sourcen, d.h. unterschiedliche Quellcodes fuer sun4 und i86pc. Deshalb ist es i.A. nicht noetig in den Variantenverzeichnissen sun4 oder i86pc irgendetwas zu machen. CB_Make verzweigt automatisch in das richtige Varianten- verzeichnis (z.B. von serverSources/Prolog_Files in sun4/serverSources/Prolog_Files) mit den notwendigen Parametern. Ein manueller CB_Make-Aufruf in einem Variantenverzeichnis wird wahrscheinlich nicht das erwuenschte Ergebnis bringen, weil die Parameter falsch sind.

Folgende Kommandos sollten in allen Verzeichnissen funktionieren:

  • CB_Make (ohne Parameter) erstellt im aktuellen Verzeichnis und in allen Subdirectories die Default-Komponenten.
  • CB_Make info gibt einige Informationen ueber das aktuelle Pool-Verzeichnis aus. Dort sind z.B. alle Subdirectories und Source-Files aufgelistet.
  • CB_Make <DIR> erstellt das Subdirectory DIR und legt die notwendigen Links auf den globalen Pool an, wenn DIR bei CB_Make info als Subdirectory aufgelistet wurde.
  • CB_Make clean loescht die Dateien, die mit CB_Make erstellt worden sind, also *.o *.wic lib*.a usw. Falls Subdirectories vorhanden sind, wird in diesen Verzeichnissen ebenfalls CB_Make clean aufgerufen.
  • CB_Make export kopiert die Dateien, die nachher mitverteilt werden sollen oder irgendwie global verwendet werden koennen, in ein Verzeichnis EXPORT. Fuer einen lokalen Pool macht das nicht viel Sinn, im globalen Pool werden spaeter diese Dateien nach CB_Product kopiert.
  • CB_Make depend erstellt zu C- und C++-Dateien, die Dependencies.
  • CB_Make poolstruct erstellt rekursiv alle Unterverzeichnisse des aktuelle Verzeichnisses.

Spezielle Kommandos fuer das Verzeichnis ProductPOOL/serverSources:

  • CB_Make iserver startet einen interaktiven Server. Dieses Kommando braucht man eigentlich nicht. Das Skript CBserver ruft CB_Make iserver auf, wenn der Parameter -i benutzt wird. Damit das Skript den lokalen Pool auch findet, muss die Umgebungsvariable CB_WORK auf das Verzeichnis fuer den Pool gesetzt sein, also z.B. mit "setenv CB_WORK /HOME2/CBASE/quix/CB/CB_Work/ProductPOOL".
  • CB_Make link_server linkt einen Server. cbRTexec und cbRTdata liegen dann unter <variante>/serverSources/objects.

    CB_Make iserver und link_server benutzen Dateien aus dem lokalen Pool, falls diese dort vorhanden sind und ansonsten aus dem globalen Pool. Vor dem Start/Linken des Servers werden diese lokalen Dateien nochmals uebersetzt. Wer Prolog-Dateien beim interaktiven Server reconsulten moechte, ruft mit Prolog-Call das Praedikat reconsult/0 auf, es werden dann alle lokalen Prolog-Dateien nochmal uebersetzt und nachgeladen. Dies geht allerdings nicht mit allen Prolog-Dateien.
